# T08.1 - Deepen moe router top-k behavior

## Context

The router exists as a foundation, but Sprint 08 expects real top-k selection
and load-balance semantics.

## Acceptance criteria

- [x] Router exposes top-k selection and entropy/load-balance signals
- [x] Contracts protect expert selection behavior
- [x] Telemetry surface is explicit

## Current progress

- `runtime/moe/router.cpp` now computes softmax-backed top-k routing and keeps
  `entropy`, `load_balance`, `selected_mass`, and `total_experts` in a
  retained `RouterDecision`
- MoE adapters surface router telemetry in native generation output via
  `moe_selected_experts`, `moe_router_entropy`, `moe_load_balance`, and
  `moe_selected_mass`
- Native regression now protects both router internals and visible DeepSeek
  CLI telemetry
