# T04.1 - Expand neon matmul hot path coverage

## Context

The repo already exposes `fp32`, `fp16`, and `bf16` NEON intent, but the
matmul path still needs stronger vectorized coverage and a real `int8-dot`
execution path to match Sprint 04 goals.

## Acceptance criteria

- [ ] `runtime/neon/neon_matmul.cpp` covers `fp16`, `bf16`, and `int8-dot`
      execution paths with host-aware fallback
- [ ] Planner flavor and executed kernel stay aligned in runtime telemetry
- [ ] Native contracts protect parity versus the scalar baseline

## Validation

- build/lint/unit
- native contract runner
- regression against scalar path
- benchmark evidence in `runtime/benchmarks/dense_baseline.cpp`
